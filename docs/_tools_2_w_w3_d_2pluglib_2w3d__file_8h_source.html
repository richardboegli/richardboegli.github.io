51VERSION NUMBERS:
52
53 Each Major chunk type will contain a "header" as its first
54 sub-chunk. The first member of this header will be a Version
55 number formatted so that its major revision number is the
56 high two bytes and its minor revision number is the lower two
57 bytes.
58
59Version 1.0:
60
61 MESHES - contained the following chunks:
62 W3D_CHUNK_MESH_HEADER, // header for a mesh
63 W3D_CHUNK_VERTICES, // array of vertices
64 W3D_CHUNK_VERTEX_NORMALS, // array of normals
65 W3D_CHUNK_SURRENDER_NORMALS, // array of surrender normals (one per vertex as req. by surrender)
66 W3D_CHUNK_TEXCOORDS, // array of texture coordinates
67 W3D_CHUNK_MATERIALS, // array of materials
68 W3D_CHUNK_TRIANGLES, // array of triangles
69 W3D_CHUNK_SURRENDER_TRIANGLES,// array of surrender format tris
70 W3D_CHUNK_MESH_USER_TEXT, // Name of owning hierarchy, text from the MAX comment field
71
72 HIERARCHY TREES - contained the following chunks:
73 W3D_CHUNK_HIERARCHY_HEADER,
74 W3D_CHUNK_PIVOTS,
75 W3D_CHUNK_PIVOT_FIXUPS,
76
77 HIERARCHY ANIMATIONS - contained the following chunks:
78 W3D_CHUNK_ANIMATION_HEADER,
79 W3D_CHUNK_ANIMATION_CHANNEL,
80
81 MESH CONNECTIONS - (blueprint for a hierarchical model) contained these chunks:
82

270/********************************************************************************
271
272 WTM ( Westwood Triangle Mesh )
273
274 Each mesh will be contained within a WTM_CHUNK_MESH within
275 this chunk will be the following chunks:
276
277 The header will be the first chunk and it tells general
278 information about the mesh such as how many triangles there
279 are, how many vertices, the bounding box, center
280 of mass, inertia matrix, etc.
281
282 The vertex array is an array of Vectors giving the object
283 space location of each vertex
284
285 The normal array is an array of all of the unique vertex
286 normal vectors needed by the mesh. This allows for vertices
287 with multiple normals so that we can duplicate the effect
288 of the smoothing groups in 3dsMax.
289
290 The surrender normal array is an array of vertex normals which
291 correspond 1to1 with the vertices. This is because the current
292 version of surrender can only handle one vertex normal per vertex.
293 In this case, the application should skip the normal array chunk
294 and read the surrender normal chunk into its SR_VERTs
295
296 The texture coord array is all of the unique texture coordinates
297 for the mesh. This allows triangles to share vertices but not
298 necessarily share texture coordinates.
299
300 The material array is a list of the names and rgb colors of
301 all of the unique materials used by the mesh. All triangles will
302 have a material index into this list.
303
304 The triangle array is all of the triangles which make up the
305 mesh. Each triangle has 3 indices to its vertices, vertex normals,
306 and texture coordinates. Each also has a material id and the
307 coefficients for its plane equation.
308
309 The Surrender Triangle array is all of the triangles in a slightly
310 different format. Surrender triangles contain their u-v coordinates
311 so there is no indirection and no possibility for sharing. To
312 make the importer faster, the triangles will also be stored in this
313 format. The application can read whichever chunk it wants to.
314
315 The mesh user text chunk is a NULL-terminated text buffer.
316
317********************************************************************************/

426// Version 3.0 Material, A W3D_CHUNK_MATERIALS3 chunk will wrap a bunch of
427// W3D_CHUNK_MATERIAL3 chunks. Inside each chunk will be a name chunk, an 'info' chunk which
428// contains the following struct, and one or more map chunks. a mesh with 2 materials might
429// look like:
430//
431// W3D_CHUNK_MATERIALS3 <-- simply a wrapper around the array of mtls
432// W3D_CHUNK_MATERIAL3 <-- a wrapper around each material
433// W3D_CHUNK_STRING <-- name of the material
434// W3D_CHUNK_MATERIAL3_INFO <-- standard material properties, a W3dMaterial3Struct
435// W3D_CHUNK_MATERIAL3_DC_MAP <-- a map, W3dMap3Struct
436// W3D_CHUNK_STRING <-- filename of the map
437// W3D_CHUNK_MAP_INFO <-- map parameters
438// W3D_CHUNK_MATERIAL3_SC_MAP
439// W3D_CHUNK_STRING <-- filename of the map
440// W3D_CHUNK_MAP_INFO
441// W3D_CHUNK_MATERIAL3
442// W3D_CHUNK_MATERIAL3_NAME
443// W3D_CHUNK_MATERIAL3_INFO
444// W3D_CHUNK_MATERIAL3_SI_MAP
445//
447struct W3dMaterial3Struct
448{
449 uint32 Attributes; // flags,hints,etc.
450
451 W3dRGBStruct DiffuseColor; // diffuse color
452 W3dRGBStruct SpecularColor; // specular color
453
454 W3dRGBStruct EmissiveCoefficients; // emmissive coefficients, default to 0,0,0
455 W3dRGBStruct AmbientCoefficients; // ambient coefficients, default to 1,1,1
456 W3dRGBStruct DiffuseCoefficients; // diffuse coeficients, default to 1,1,1
457 W3dRGBStruct SpecularCoefficients; // specular coefficients, default to 0,0,0
458
459 float32 Shininess; // how tight the specular highlight will be, 1 - 1000 (default = 1)
460 float32 Opacity; // how opaque the material is, 0.0 = invisible, 1.0 = fully opaque (default = 1)
461 float32 Translucency; // how much light passes through the material. (default = 0)
462 float32 FogCoeff; // effect of fog (0.0=not fogged, 1.0=fogged) (default = 1)
463};

667/********************************************************************************
668
669 WHT ( Westwood Hierarchy Tree )
670
671 A hierarchy tree defines a set of coordinate systems which are connected
672 hierarchically. The header defines the name, number of pivots, etc.
673 The pivots chunk will contain a W3dPivotStructs for each node in the
674 tree.
675
676 The W3dPivotFixupStruct contains a transform for each MAX coordinate
677 system and our version of that same coordinate system (bone). It is
678 needed when the user exports the base pose using "Translation Only".
679 These are the matrices which go from the MAX rotated coordinate systems
680 to a system which is unrotated in the base pose. These transformations
681 are needed when exporting a hierarchy animation with the given hierarchy
682 tree file.
683
684 Another explanation of these kludgy "fixup" matrices:
685
686 What are the "fixup" matrices? These are the transforms which
687 were applied to the base pose when the user wanted to force the
688 base pose to use only matrices with certain properties. For
689 example, if we wanted the base pose to use translations only,
690 the fixup transform for each node is a transform which when
691 multiplied by the real node's world transform, yeilds a pure
692 translation matrix. Fixup matrices are used in the mesh
693 exporter since all vertices must be transformed by their inverses
694 in order to make things work. They also show up in the animation
695 exporter because they are needed to make the animation work with
696 the new base pose.
697
698********************************************************************************/
