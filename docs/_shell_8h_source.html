25// FILE: Shell.h //////////////////////////////////////////////////////////////////////////////////
26// Author: Colin Day, September 2001
27// Description: Shell menu representations
28//
29// Using The Shell:
30// ----------------
31//
32// The Shell makes use of the window layouts to represent screens. You can push and
33// pop screens on the stack so that you don't have to keep track of how you got
34// to which screen.
35//
36// Here is what happens when you push a layout on the shell stack:
37// 1) The layout name is stored as a "pending push"
38// 2) The top of the stack has the Shutdown() method run
39// 3) The Shutdown() method (or other mechanisms like Update()) will call
40// Shell::shutdownComplete() when the shutdown process for that layout is "complete"
41// 4) Shell::shutdownComplete() sees the "pending push" in the shell
42// 5) "Pending push" layout is then loaded from disk and the pending push state is cleared
43// 6) The new layout is put on the top of the stack
44// 7) The new layout Init() method is called
45//
46// Here is what happens when you pop the top of the stack
47// 1) The stack sets a "pending pop" as in progress
48// 2) The top layout of the stack has the Shutdown() method called
49// 3) The Shutdown() method (or other mechanisms like Update()) will call
50// Shell::shutdownComplete() when the shutdown process for that layout is "complete"
51// 4) Shell::shutdownComplete() sees the "pending pop" in the shell
52// 5) The "pending pop" layout is then destroyed and removed from the stack
53// 6) The new top of the stack has the Init() method run
54//
55// Window Layouts and the Shell:
56// -----------------------------
57//
58// Window Layouts in the shell need the following functions to work property, these
59// can be assigned from the GUIEdit window layout tool:
60//
61// - Init() [OPTIONAL]
62// This is called as a result of a push or pop operation (see above for more info)
63// The window layout is loaded from disk and then this Init()
64// method is run. All shell layout Init() methods should show
65// the layout windows. At this point you could move windows
66// to starting positions, set a state that the Update() method looks at to
67// "animate" the windows to the desired positions.
68//
69// - Update() [OPTIONAL]
70// This is called once at a rate of "shellUpdateDelay" for EVERY screen on the shell
71// stack. It does not matter if the screen is on the top, or is hidden, or
72// anything, this is always called. Each update is run starting with the screen
73// at the top of the stack and progressing to the bottom of the stack.
74// States could be set in the Init() or Shutdown() methods of the layout
75// that the Update() looks at and reacts to appropriately if desired.
76//
77// - Shutdown() [REQUIRED]
78// This is called when a layout is popped off the stack, or when a new layout
79// is pushed on top of this one (see above for more detail on what happens
80// during the push/pop process). You can switch into a "shutdown" state and
81// animate the layout appropriately in the Update() method for the layout.
82// When shutdown is actually complete you should hide the all windows in
83// the layout and then you are REQUIRED to notify the shell by calling
84// the Shell::shutdownComplete() method.
85//
86// Shutdown() is also required to be able to handle the paramater "immediatePop".
87// If this paramater is TRUE it means that when control returns from the
88// shutdown function that the layout will immediately be popped off the
89// stack. We need to be able to handle this when in code we want to
90// traverse back down the stack rapidly (like when we lose connection to
91// an online service, we might pop all the way back to the login screen)
92//
